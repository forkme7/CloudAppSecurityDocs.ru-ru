# <a name="investigate"></a>Анализ
После запуска Cloud App Security в облачной среде потребуется произвести исследование и анализ с помощью средств Cloud App Security, чтобы лучше понять, что именно происходит в этой среде. Затем, основываясь на особенностях конкретной среды и ее использовании, можно определить требования, необходимые для защиты организации от рисков.  
  
Этот раздел описывает проведение глубокого анализа, помогающего лучше понять суть происходящего в облачной среде.  
  
## <a name="dashboards"></a>Панели мониторинга  
Следующие панели мониторинга помогут вам изучить работу приложений в облачной среде:  
  
|Панель мониторинга|Описание|  
|---------------|-----------------|  
|Основная панель мониторинга|Общие сведения о состоянии облака (пользователи, файлы и действия), а также необходимых действиях (оповещения, нарушения действий и содержимого)|  
|Панель мониторинга приложений — общие сведения|Обзор использования приложений в каждом месте, графики использования для нескольких пользователей|  
|Панель мониторинга приложений — ценные сведения|Анализ данных, хранящихся в приложении; разбивка по типу файлов и уровню совместного использования|  
|Панель мониторинга приложений — файлы|Детализация файлов, возможность фильтрации по владельцу, уровню совместного использования и т. д., а также выполнение действий системы управления (например, помещение в карантин)|  
|Панель мониторинга приложений — сторонние приложения|Детализация уже развернутых сторонних приложений, таких как Google Apps, определение политик для них|  
|Панель мониторинга пользователя|Полный обзор профиля пользователя в облаке, включая группы, расположения, последние действия, связанные оповещения и использованные браузеры|  
  
##  <a name="a-namesanctionappa-sanction-or-unsanction-apps"></a> Применение или отмена санкционирования приложений  
Первый шаг к пониманию облака заключается в санкционировании приложений. После санкционирования приложения можно отфильтровать несанкционированные приложения и запустить миграцию на санкционированные приложения аналогичного типа.  
  
-   В консоли Cloud App Security щелкните **Обнаружение**, а затем — **Панель мониторинга обнаружения**.  
  
-   В списке обнаруженных приложений щелкните многоточие ![многоточие в строке приложения](./media/sanction-three-dots.png "Sanction three dots") в конце строки с приложением, которое нужно санкционировать, и выберите **Пометить как санкционированную**.  
  
     ![пометить как санкционированную](./media/mark-as-sanctioned.png "mark as sanctioned")  
  
> [!NOTE]  
>  Для каждого приложения, которое необходимо отслеживать с помощью интеграции API Cloud App Security, рекомендуется создать в Cloud App Security выделенную учетную запись службы с правами администратора.  
  
## <a name="use-the-investigation-tools"></a>Использование средств для исследования  
  
1.  На портале Cloud App Security выберите **Сведения** и просмотрите **Журнал действий** и фильтр для конкретного приложения. Проверьте соблюдение перечисленных ниже условий.  
  
    -   Кто получает доступ к облачной среде?  
  
    -   Из каких диапазонов IP-адресов?  
  
    -   Что делает администратор?  
  
    -   Из каких расположений подключаются администраторы?  
  
    -   Подключаются ли к облачной среде какие-либо устаревшие устройства?  
  
    -   Неудачные попытки входа поступают с ожидаемых IP-адресов?  
  
2.  Выберите **Сведения**, **Файлы** и проверьте следующее:  
  
    -   Сколько файлов открыто для общего доступа, так что любой пользователь может обратиться к ним без ссылки?  
  
    -   С какими партнерами вы осуществляете общий доступ к файлам? (исходящий общий доступ)  
  
    -   Имеются ли файлы, содержащие конфиденциальное имя?  
  
    -   Используется ли чья-то личная учетная запись для общего доступа к файлам?  
  
3.  Выберите **Сведения**, **Учетные записи** и проверьте следующее:  
  
    -   Существуют ли учетные записи, которые уже длительное время остаются неактивными в определенной службе? Возможно, следует отменить у такого пользователя лицензию на эту службу?  
  
    -   Хотите ли вы знать, какие пользователи имеют определенную роль?  
  
    -   Был ли уволен сотрудник, который мог сохранить доступ к приложению и использовать его для кражи информации?  
  
    -   Хотите ли вы отменить разрешение пользователя на определенное приложение или потребовать от конкретного пользователя прохождения многофакторной проверки подлинности?  
  
4.  Выберите **Сведения** и затем приложение. Открывается панель мониторинга приложения с информацией и ценными сведениями. Вкладки в верхней ее части можно использовать для ответа на следующие вопросы:  
  
     ![анализ приложения](./media/investigate-app.png "investigate app")  
  
    -   Какие устройства используют пользователи для подключения к приложению?  
  
    -   Какие типы файлов они сохраняют в облаке?  
  
    -   Какие действия выполняются в приложении прямо сейчас?  
  
    -   Существуют ли в вашей среде подключенные сторонние приложения?  
  
    -   Известно ли вам об этих приложениях?  
  
    -   Авторизованы ли они для предоставленного им уровня доступа?  
  
    -   У скольких пользователей они развернуты? Насколько часто они встречаются?  
  
5.  Перейдите на **панель мониторинга Cloud Discovery** и проверьте следующее:  
  
    -   Какие облачные приложения используются, в какой степени и кем?  
  
    -   Для каких целей они используются?  
  
    -   Какой объем данных отправляется в эти облачные приложения?  
  
    -   В каких категориях у вас присутствуют санкционированные облачные приложения, но пользователи работают с альтернативными решениями?  
  
    -   Имеются ли у альтернативного решения облачные приложения, для которых вы хотите отменить санкционирование в своей организации?  
  
    -   Существуют ли используемые облачные приложения, которые не соответствуют политике вашей организации?  
  
## <a name="how-to-use-reports-to-investigate-risk"></a>Использование отчетов для исследования рисков  
Начиная управлять облачной средой, вы делаете определенные предположения, исходя из ожидаемых результатов, так как еще мало знаете о своем облаке. И на основе этих допущений вы создаете политики. После запуска Cloud App Security в облачной среде вы можете использовать встроенные отчеты (а также настраиваемые отчеты), чтобы узнать, что именно происходит в облаке. Основываясь на полученных сведениях, вы настраиваете политики еще раз, чтобы учесть исключения. В итоге политика выдает крайне мало ложных положительных результатов.  
  
Встроенные отчеты предоставляют агрегированные представления для изучения.  
  
Для работы со встроенными отчетами выберите **Сведения**, а затем — **Встроенные отчеты**. Дополнительные сведения о различных встроенных отчетах см. в статье [Справочник по встроенным отчетам](built-in-report-reference.md).  
  
## <a name="sample-investigation"></a>Пример исследования  
Предположим, вы считаете, что к вашей облачной среде не осуществляется доступ с опасных IP-адресов (например, с анонимных прокси-серверов и Tor). Но на всякий случай вы создаете соответствующую политику Risky IP:  
  
1.  На портале выберите **Управление** и затем **Политики**.  
  
2.  В **Центре политик** откройте вкладку **Шаблоны**.  
  
3.  В конце строки **Вход пользователя с IP-адреса, не имеющего категории** щелкните **+** для создания политики.  
  
4.  Измените имя политики, чтобы вы могли узнать ее.  
  
5.  В разделе **Фильтры событий** щелкните **+** для добавления фильтра. Прокрутите вниз до элемента **Тег IP-адреса**, а затем выберите **Анонимный** и **Tor**.  
  
     ![пример политики в отношении рискованных ip-адресов](./media/example-policy-risky-ips.png "example policy risky ips")  
  
Теперь, когда политика готова, вы с удивлением получаете оповещение о нарушении политики.  
  
1.  Перейдите на страницу **Оповещения** и просмотрите оповещение о нарушении политики.  
  
2.  Если это действительно нарушение, следует устранить его или снизить риск.  
  
     Чтобы снизить риск, можно отправить пользователю уведомление с вопросом о том, было ли нарушение намеренным и знал ли о нем пользователь.  
  
     Можно также детализировать оповещение и приостановить действие учетной записи пользователя до выяснения обстоятельств и требуемых мер.  
  
3.  Если это допустимое событие, которое вряд ли повторится, оповещение можно проигнорировать.  
  
     Если это допустимое событие, которое, возможно, будет повторяться, можно изменить политику, чтобы в будущем подобное событие не считалось нарушением.
