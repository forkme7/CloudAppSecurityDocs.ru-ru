# <a name="anomaly-detection-policy"></a>Политика обнаружения аномалий
В этой статье описываются справочные сведения о политиках, содержащие пояснения по каждому типу политик и полям, которые можно настроить для каждой политики.  

После защиты организации с помощью Cloud App Security все действия в облаке оцениваются в соответствии с различными предопределенными факторами риска. Cloud App Security проверяет каждый сеанс пользователя в облаке и затем учитывает заданные здесь факторы риска, чтобы оповестить вас о событии, которое отличается от базовых показателей организации или от типичных действий пользователей. На странице политики обнаружения аномалий можно настроить, какие семейства факторов риска следует учитывать при оценке риска. Политики могут применяться по-разному для различных пользователей, расположений и секторов организации. Например, можно создать политику, оповещающую вас о деятельности членов ИТ-группы за пределами офисов.  

Служба Cloud App Security имеет начальный период обучения длительностью в 7 дней, в течение которого она не отмечает новых пользователей, действия, устройства или расположения как аномальные. После этого каждый сеанс сравнивается с действиями, периодами активности, IP-адресами, устройствами и другими данными, зарегистрированными за последний месяц, а также оценкой риска этих действий. С помощью ползунка чувствительности можно задать для политики минимальный уровень риска для активации оповещений. При создании политики обнаружения аномалий рекомендуется использовать порог чувствительности по умолчанию в течение одной недели, прежде чем менять его в соответствии с числом полученных оповещений. Служба Cloud App Security будет отправлять вам разное число оповещений для разных уровней угроз при изменении порога чувствительности.
  
![ползунок чувствительности](./media/sensitivity-slider.png)

Настройка политики обнаружения аномалий:  
  
1.  В консоли выберите элемент **Управление** и затем **Политики**.  
  
2.  Щелкните **Создание политики** и выберите политику **Обнаружение аномалий**.  
  
     ![меню политики обнаружения аномалий](./media/anomaly-detection-policy-menu.png "меню политики обнаружения аномалий")  
  
3.  Введите имя и описание политики и перейдите к полю **Фильтры событий**, где можно выбрать действие, для которого требуется применить политику.  
  
4.  Назначьте политике имя и описание. При необходимости можно использовать в качестве основы политики шаблон. Дополнительные сведения о шаблонах политик см. в статье [Управление облачными приложениями с помощью политик](control-cloud-apps-with-policies.md).  
  
5.  Чтобы применить политику ко всем действиям в облачной среде, выберите **Все отслеживаемые действия**. Чтобы ограничить область действия политики отдельными типами действий, выберите **Выбранное действие**. Щелкните **Добавить фильтры** и задайте параметры для фильтрации действия. Например, чтобы применить политику только к действию, выполненному администраторами Salesforce, выберите этот тег пользователя.  
  
6.  Под этим полем задайте **Факторы риска**. Можно выбрать, какие семейства риска требуется учитывать при оценке рисков. Можно использовать кнопку включения и отключения в правой части строки, чтобы включать и отключать разные риски. Кроме того, для большей детальности можно выбрать действие, для которого необходимо включить отдельные семейства рисков.  
  
     Существуют следующие факторы риска:  
  
    -   **Ошибки входа**: предпринял ли пользователь несколько неудачных попыток войти в систему за короткий период?  
  
    -   **Действия администратора**: используют ли администраторы свои привилегированные учетные записи для входа из нестандартных расположений или в необычное время?  
  
    -   **Неактивные учетные записи**: возобновилась ли активность учетной записи, которая некоторое время оставалась неактивной?  
  
    -   **Расположение**: выполняются ли действия из необычного, подозрительного или нового расположения?  
  
    -   **Неосуществимое перемещение**: входит ли пользователь в систему из Москвы, а через 10 минут — из Парижа?  
  
    -   **Устройство и агент пользователя**: выполняются ли действия с нераспознанных или неуправляемых устройств?  

    -   **Частота действия**: не возникает ли внезапно скачка активности у конкретного приложения? Это может быть большой объем скачиваемых или удаляемых данных, публикация большого числа файлов или большое число административных действий.
  
     Эти параметры можно использовать для определения сложных сценариев, например, чтобы исключить диапазон IP-адресов офиса из учитываемых факторов риска для обнаружения аномалий, создайте отдельный тег "office IP" и отфильтруйте диапазон из учитываемых параметров. Чтобы исключить созданный диапазон из процедуры обнаружения аномалий в действиях администратора, сделайте следующее:  
  
    -   В разделе **Тип риска** найдите **Активность администратора**.  
  
    -   Измените значение параметра **Применить к** на **Выбранное действие**.  
  
    -   В разделе **Фильтры событий** задайте для параметра **Применить к** значение **Выбранное действие**, а в разделе **Действия, соответствующие всем следующим условиям** выберите для параметра **Административное действие** значение **True**.  
  
    -   Щелкните значок **+**, выберите **Тег IP-адреса не равно** и тег "Office IP".  
  
7.  В разделе **Чувствительность** выберите, как часто вы хотите получать оповещения.  
  
     Значение чувствительности позволяет определить, сколько еженедельных оповещений активируется в среднем на каждую 1000 пользователей.  
  
     ![IP-адреса обнаружения аномалий](./media/anomaly-detection-ips.png "IP-адреса обнаружения аномалий")  
  
8.  Нажмите кнопку **Create** (Создать).  
 

## <a name="anomaly-detection-policy-reference"></a>Справочник по политикам обнаружения аномалий  
Политика обнаружения аномалий позволяет настроить постоянный мониторинг действий пользователей для определения аномального поведения. Аномалии определяются путем проверки активности пользователя. Риск оценивается по более чем 30 различным индикаторам риска, сгруппированным по шести факторам риска. На основании результатов политики активируются оповещения безопасности.   
Каждая политика состоит из следующих элементов:  
  
-   Фильтры действий — позволяют выборочно сканировать на аномалии только отфильтрованную деятельность пользователя.  
  
-   Факторы риска — позволяют выбрать, какие факторы риска следует учитывать при оценке рисков.  
  
-   Чувствительность — позволяет задать количество активируемых политикой оповещений.  
  
### <a name="activity-filters"></a>Фильтры действий  
Список фильтров действий см. в разделе [добавить описание ссылки](activity-filters.md).  
  
### <a name="risk-factors"></a>Факторы риска  
Ниже приведен список факторов риска, учитываемых при оценке риска действий пользователей. Каждый фактор риска можно включить или отключить. Для каждого фактора риска в поле **Применить к** можно указать два параметра, определяющих, следует ли включить этот фактор в оценку рисков действий пользователей:  
  
-   "Все отслеживаемые действия" — все действия пользователей, отбираемые фильтром действий политики.  
  
-   "Выбранное действие" — действия пользователей, отбираемые фильтром действий политики и фильтром действий для этого фактора риска. При выборе этого параметра под фактором риска отображается средство выбора фильтра действий, которое работает точно так же, как фильтр действий политики.  
  
Каждый фактор риска при включении в оценку рисков привносит собственные триггеры, что ведет к увеличению оцениваемого уровня риска для активности пользователей:  
  
-   "Ошибки входа" — большое количество ошибок входа или активность, полностью состоящая из ошибок входа.  
  
-   "Активность администратора" — административные действия, выполненные неучтенным пользователем либо с нового IP-адреса, из расположения или от поставщика услуг Интернета, которые не используются корпоративными пользователями.  
  
-   "Неактивные учетные записи" — действия, выполняемые пользователем, который длительное время оставался неактивным.  
  
-   "Расположение" — действия, выполняемые с IP-адреса, от поставщика услуг Интернета или из расположения, которые никогда не использовались, не использовались другими пользователями или определенным пользователем либо же использовались только для неудачных попыток входа.  
  
-   "Неосуществимое перемещение" — действия из разных удаленных расположений за короткое время.  
  
-   "Устройство и агент пользователя" — действия, выполняемые с помощью пользовательского агента или устройства, которые никогда не использовались другими пользователями или определенным пользователем.  
  
-   "Частота действий" — повторяющиеся действия пользователя за короткое время. 

### <a name="sensitivity"></a>Чувствительность  
Управлять числом оповещений, активируемых политикой, можно двумя способами:  
  
-   Ползунок "Чувствительность" — выберите, сколько оповещений следует активировать на 1000 пользователей в неделю. Оповещения будут создаваться для действий с наивысшим уровнем риска.  
  
-   "Ежедневный лимит оповещений" — ограничьте число оповещений, выдаваемых в течение одного дня.
