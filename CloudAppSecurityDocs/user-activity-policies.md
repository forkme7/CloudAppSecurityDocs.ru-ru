# <a name="user-activity-policies"></a>Политики действий пользователей
Политики действий позволяют применять разнообразные автоматизированные процессы, используя API поставщика приложения. Эти политики позволяют отслеживать определенные действия, выполняемые различными пользователями, либо неожиданно высокую частоту одного определенного типа действия.  
  
После задания политики обнаружения действий она начинает создавать оповещения. Оповещения создаются только для действий, которые произошли после создания политики.
  
  
## <a name="custom-alerts"></a>Настраиваемые оповещения  
Политики действий позволяют задавать настраиваемые оповещения или действия, отправляемые или выполняемые при обнаружении действий пользователей. Например, если вы хотите знать о каждом случае, когда пользователь неудачно пытается войти в систему 70 раз за одну минуту, скачивает 7000 файлов или выполнил вход из Афганистана, можно задать оповещения действий, отправляемые вам или соответствующему пользователю при возникновении таких событий. Можно даже приостановить действие учетной записи пользователя на время расследования.  
  
Чтобы создать политику действий, выполните следующую процедуру:  
  
1.  В консоли выберите элемент **Управление** и затем **Политики**.  
  
2.  Щелкните **Создание политики** и выберите **Политика действий**.  
  
     ![меню политики действий](./media/activity-policy-menu.png "activity policy menu")  
  
3.  Назначьте политике имя и описание. При необходимости можно использовать в качестве основы политики шаблон. Дополнительные сведения о шаблонах политик см. в статье [Управление облачными приложениями с помощью политик](control-cloud-apps-with-policies.md).  
  
4.  Чтобы задать действия или другие метрики, активируемые этой политикой, используйте **фильтры действий**.  
  
5.  В разделе **Параметры соответствия действий** укажите, возникает ли нарушение политики, когда только одно действие соответствует фильтрам, или для этого требуется обнаружить указанное **Число повторяющихся действий**.  
    Если выбрать **Повторить действие**, вы можете задать **действия, соответствующие группам, для приложения**. Это активизирует сопоставление политики только в том случае, если повторяющиеся действия выполняются в том же приложении (например, 5 загрузок из Box).  
  
6.  Настройте **Действия**, которые требуется выполнить при обнаружении совпадения.  
  
Рассмотрим следующие примеры:  
  
-   Несколько неудачных попыток входа  
  
     Можно настроить политику для выдачи оповещения при наличии большого количества неудачных попыток входа в течение небольшого периода. Для этого выберите подходящий фильтр действий на странице **Новая политика действия**.  
  
     Под полем **Фильтры событий** настройте параметры, для которых будет активироваться оповещение.  
  
     ![пример политики для неудачных попыток входа](./media/multiple-failed-log-on-attempts-policy-example.png "multiple failed log on attempts policy example")  
  
-   Высокая частота скачиваний  
  
     Можно настроить политику для выдачи оповещения при наличии непредвиденного или нехарактерного уровня скачиваний. Для этого в разделе **Скорость** выберите параметры для активации оповещения.  
  
     ![пример высокой частоты скачиваний](./media/high-download-rate-example.png "high download rate example")  
  
## <a name="anomaly-detection"></a>Обнаружение аномалий  
После защиты организации с помощью Cloud App Security все действия в облаке оцениваются в соответствии с различными предопределенными факторами риска. Cloud App Security проверяет каждый сеанс пользователя в облаке и затем учитывает заданные здесь факторы риска, чтобы оповестить вас о событии, которое отличается от базовых показателей организации или от типичных действий пользователей. На странице политики обнаружения аномалий можно настроить, какие семейства факторов риска будут учитываться при оценке риска. Политики могут применяться по-разному для различных пользователей, расположений и секторов организации. Например, можно создать политику, оповещающую вас о деятельности членов ИТ-группы за пределами офисов.  
  
Настройка политики обнаружения аномалий:  
  
1.  В консоли выберите элемент **Управление** и затем **Политики**.  
  
2.  Щелкните **Создание политики** и выберите политику **Обнаружение аномалий**.  
  
     ![меню политики обнаружения аномалий](./media/anomaly-detection-policy-menu.png "Anomaly detection policy menu")  
  
3.  Введите имя и описание политики и перейдите к полю **Фильтры событий**, где можно выбрать действие, для которого требуется применить политику.  
  
4.  Назначьте политике имя и описание. При необходимости можно использовать в качестве основы политики шаблон. Дополнительные сведения о шаблонах политик см. в статье [Управление облачными приложениями с помощью политик](control-cloud-apps-with-policies.md).  
  
5.  Чтобы применить политику ко всем действиям в облачной среде, выберите **Все отслеживаемые действия**. Чтобы ограничить область действия политики отдельными типами действий, выберите **Выбранное действие**. Щелкните **Добавить фильтры** и задайте параметры для фильтрации действия. Например, чтобы применить политику только к действию, выполненному администраторами Salesforce, выберите этот тег пользователя.  
  
6.  Под этим полем задайте **Факторы риска**. Можно выбрать, какие семейства риска требуется учитывать при оценке рисков. Можно использовать кнопку включения и отключения в правой части строки, чтобы включать и отключать различные риски. Кроме того, для большей детальности можно выбрать действие, для которого необходимо включить отдельные семейства рисков.  
  
     Существуют следующие факторы риска:  
  
    -   **Ошибки входа**: предпринял ли пользователь несколько неудачных попыток войти в систему за короткий период?  
  
    -   **Действия администратора**: используют ли администраторы свои привилегированные учетные записи для входа из нестандартных расположений или в необычное время?  
  
    -   **Неактивные учетные записи**: возобновилась ли активность учетной записи, которая некоторое время оставалась неактивной?  
  
    -   **Расположение**: выполняются ли действия из необычного, подозрительного или нового расположения?  
  
    -   **Неосуществимое перемещение**: входит ли пользователь в систему из Москвы, а через десять минут — из Парижа?  
  
    -   **Устройство и агент пользователя**: выполняются ли действия с нераспознанных или неуправляемых устройств?  
  
     Эти параметры можно использовать для определения сложных сценариев, например, чтобы исключить диапазон IP-адресов офиса из учитываемых факторов риска для обнаружения аномалий, создайте отдельный тег "office IP" и отфильтруйте диапазон из учитываемых параметров. Чтобы исключить созданный диапазон из процедуры обнаружения аномалий в действиях администратора, выполните следующее:  
  
    -   В разделе **Тип риска** найдите **Активность администратора**.  
  
    -   Измените значение параметра **Применить к** на **Выбранное действие**.  
  
    -   В разделе **Фильтры событий** задайте для параметра **Применить к** значение **Выбранное действие**, а в разделе **Действия, соответствующие всем следующим условиям** выберите для параметра **Административное действие** значение **True**.  
  
    -   Щелкните значок **+**, выберите **Тег IP-адреса не равно** и тег "Office IP".  
  
7.  В разделе **Чувствительность** выберите, как часто вы хотите получать оповещения.  
  
     Значение чувствительности определит, сколько еженедельных оповещений активируется в среднем на каждую 1000 пользователей.  
  
     ![IP-адреса обнаружения аномалий](./media/anomaly-detection-ips.png "anomaly detection IPs")  
  
8.  Нажмите кнопку **Create** (Создать).  
 
  
## <a name="activity-policy-reference"></a>Справочник по политикам действий  
В этом разделе приводятся справочные сведения о политиках, содержащие пояснения по каждому типу политик и полям, которые можно настроить для каждой политики.  
  
**Политика действий** — это основанная на API политика, позволяющая отслеживать активность вашей организации в облаке, принимая во внимание более 20 фильтров метаданных файлов (включая тип устройства и расположение). На основании результатов политики могут выдаваться уведомления, а также может приостанавливаться право пользователей на использование облачного приложения.   
Каждая политика состоит из следующих элементов:  
  
-   Фильтры событий — позволяют создавать очень детальные условия на основе метаданных.  
  
-   Параметры соответствия действий — позволяют задать пороговое значение для количества повторений действия, при котором оно считается соответствующим политике.  
  
-   Действия — политика предоставляет набор действий системы управления, которые могут применяться автоматически при обнаружении нарушений.
